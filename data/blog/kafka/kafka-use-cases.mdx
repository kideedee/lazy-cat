## 1. Consumer Stability

### 1.1 Heartbeat Model

```
┌─────────────────────────────────────────────────────────────────────────┐
│                    CONSUMER HEARTBEAT MODEL                              │
├─────────────────────────────────────────────────────────────────────────┤
│                                                                          │
│  Consumer JVM                                                            │
│  ┌─────────────────────────────────────────────────────────────────┐    │
│  │                                                                  │    │
│  │  ┌─────────────────┐         ┌─────────────────────────────┐   │    │
│  │  │ Heartbeat Thread│         │      Main Thread            │   │    │
│  │  │                 │         │                             │   │    │
│  │  │ Every 3s        │         │  poll() → process → poll() │   │    │
│  │  │ (heartbeat.     │         │                             │   │    │
│  │  │  interval.ms)   │         │  Must call poll() within    │   │    │
│  │  │                 │         │  max.poll.interval.ms       │   │    │
│  │  └────────┬────────┘         └──────────────┬──────────────┘   │    │
│  │           │                                  │                  │    │
│  └───────────┼──────────────────────────────────┼──────────────────┘    │
│              │                                  │                       │
│              ↓                                  │                       │
│  ┌───────────────────┐                          │                       │
│  │   Group Coord.    │   If no heartbeat in     │                       │
│  │   (Broker)        │   session.timeout.ms     │                       │
│  │                   │   → Consumer removed     │                       │
│  │   session.timeout │   → Rebalance triggered  │                       │
│  │   = 45s           │                          │                       │
│  └───────────────────┘                          │                       │
│              ↑                                  │                       │
│              │                                  │                       │
│              └───── If poll() not called ───────┘                       │
│                     in max.poll.interval.ms                             │
│                     → Consumer leaves group                             │
│                                                                          │
└─────────────────────────────────────────────────────────────────────────┘
```

### 1.2 Golden Rules

```
heartbeat.interval.ms ≤ session.timeout.ms / 3

max.poll.interval.ms > (max.poll.records × processing_time_per_record)
```

### 1.3 Recommended Values

| Config | Default | Production |
|--------|---------|------------|
| `session.timeout.ms` | 45000 | 30000-60000 |
| `heartbeat.interval.ms` | 3000 | session / 3 |
| `max.poll.interval.ms` | 300000 | Based on processing |
| `max.poll.records` | 500 | Tune for throughput |

### 1.4 Troubleshooting

**Problem: Consumer kicked from group**
- Log: "Member has failed to send heartbeat" hoặc "poll() was not invoked within max.poll.interval.ms"
- Root cause: Processing time > `max.poll.interval.ms`

```yaml
# Solution 1: Giảm batch size
max.poll.records: 100

# Solution 2: Tăng poll interval
max.poll.interval.ms: 600000

# Solution 3 (best): Optimize processing
# - Async processing
# - Batch database writes
# - Parallel processing
```

**Problem: Network unstable**
- Log: "Session timed out"
- Root cause: Heartbeats không đến broker

```yaml
# Tăng tolerance
session.timeout.ms: 60000
heartbeat.interval.ms: 10000
request.timeout.ms: 60000
```

**Problem: Consumer lag increasing**

```bash
# Diagnosis
kafka-consumer-groups.sh --bootstrap-server localhost:9092 \
  --describe --group my-group
```

```yaml
# Solutions
fetch.min.bytes: 50000    # Fetch nhiều hơn
fetch.max.wait.ms: 500
max.poll.records: 1000
# + Scale consumers (max = partitions)
```

## 2. Use Case 1: High Throughput

**Scenario:** Log aggregation, ETL pipelines, metrics collection

**Goal:** Maximum messages per second, accept higher latency

### 2.1 Spring Boot

```yaml
spring:
  kafka:
    producer:
      acks: 1                      # Leader only (faster)
      batch-size: 65536            # 64KB batch
      compression-type: lz4        # Fast compression
      properties:
        linger.ms: 50              # Wait to fill batch
        buffer.memory: 67108864    # 64MB buffer
    
    consumer:
      max-poll-records: 1000
      properties:
        fetch.min.bytes: 50000
        fetch.max.wait.ms: 500
```

```java
// Batch consumer
@KafkaListener(topics = "events", containerFactory = "batchFactory")
public void consumeBatch(List<Event> events, Acknowledgment ack) {
    repository.saveAll(events);  // Batch insert
    ack.acknowledge();
}
```

### 2.2 Quarkus


## 4. Use Case 3: Exactly-Once

**Scenario:** Financial transactions, inventory updates, payment processing

**Goal:** Each message processed exactly once

### 4.1 Spring Boot

```yaml
spring:
  kafka:
    producer:
      transaction-id-prefix: tx-order-service-
      properties:
        enable.idempotence: true
        acks: all
        retries: 2147483647
        max.in.flight.requests.per.connection: 5
    
    consumer:
      properties:
        isolation.level: read_committed
        enable.auto.commit: false
```

```java
@Service
@Transactional("kafkaTransactionManager")
public class OrderService {
    private final KafkaTemplate<String, Order> kafkaTemplate;
    private final OrderRepository repository;
    
    public void processOrder(Order order) {
        // Database + Kafka trong cùng transaction
        repository.save(order);
        kafkaTemplate.send("orders", order.getId(), order);
        kafkaTemplate.send("order-events", new OrderCreatedEvent(order));
        // Exception → both rollback
    }
}
```

### 4.2 Quarkus

```properties
# Producer
mp.messaging.outgoing.orders.transactional.id=tx-order-service
mp.messaging.outgoing.orders.enable.idempotence=true
mp.messaging.outgoing.orders.acks=all

# Consumer
mp.messaging.incoming.orders.isolation.level=read_committed
mp.messaging.incoming.orders.commit-strategy=throttled
mp.messaging.incoming.orders.auto.commit.interval.ms=100
```

```java
@ApplicationScoped
public class TransactionalOrderService {
    @Inject
    KafkaTransactions transactions;
    
    public Uni<Void> processOrder(Order order) {
        return transactions.withTransaction(emitter -> {
            emitter.send(order);
            emitter.send(new OrderCreatedEvent(order));
            return Uni.createFrom().voidItem();
        });
    }
}
```

## 5. Use Case 4: Event Sourcing

**Scenario:** Audit logs, CQRS, state reconstruction

**Goal:** Kafka as event store with log compaction

### 5.1 Topic Configuration

```bash
kafka-topics.sh --create \
  --topic user-events \
  --partitions 12 \
  --replication-factor 3 \
  --config cleanup.policy=compact \
  --config min.cleanable.dirty.ratio=0.3 \
  --config segment.ms=3600000 \
  --config delete.retention.ms=86400000
```

### 5.2 Implementation

```yaml
# Spring Boot
spring:
  kafka:
    producer:
      acks: all
      properties:
        enable.idempotence: true
    consumer:
      auto-offset-reset: earliest
      properties:
        isolation.level: read_committed
```

```java
public interface DomainEvent {
    String getAggregateId();
    Instant getTimestamp();
}

@Service
public class UserEventStore {
    private final KafkaTemplate<String, DomainEvent> template;
    
    public void append(DomainEvent event) {
        // Key = aggregateId → same partition → ordering
        template.send("user-events", event.getAggregateId(), event);
    }
    
    public User rebuildState(String userId) {
        // Consume từ earliest, reduce events to state
        return events.stream()
            .filter(e -> e.getAggregateId().equals(userId))
            .reduce(new User(), User::apply, (a, b) -> b);
    }
}
```

## 6. Use Case 5: CDC (Change Data Capture)

**Scenario:** Database synchronization, real-time replication

**Goal:** Capture database changes and publish to Kafka

### 6.1 Debezium Connector

```json
{
  "name": "postgres-connector",
  "config": {
    "connector.class": "io.debezium.connector.postgresql.PostgresConnector",
    "database.hostname": "postgres",
    "database.port": "5432",
    "database.user": "debezium",
    "database.password": "${POSTGRES_PASSWORD}",
    "database.dbname": "orders_db",
    "topic.prefix": "cdc",
    "table.include.list": "public.orders,public.order_items",
    "publication.autocreate.mode": "filtered",
    "slot.name": "debezium_orders",
    "transforms": "route",
    "transforms.route.type": "org.apache.kafka.connect.transforms.RegexRouter",
    "transforms.route.regex": "cdc.public.(.*)",
    "transforms.route.replacement": "cdc.$1"
  }
}
```

### 6.2 Consumer

```java
// Spring Boot
@KafkaListener(topics = "cdc.orders")
public void handleOrderChange(String payload) {
    DebeziumEvent event = parseDebeziumEvent(payload);
    
    switch (event.getOperation()) {
        case "c" -> handleCreate(event.getAfter());
        case "u" -> handleUpdate(event.getBefore(), event.getAfter());
        case "d" -> handleDelete(event.getBefore());
    }
}
```
